# 轻量级提示词流程实验台方案

## 1. 背景与定位

这个实验台用于比较不同提示词流程在同一批任务上的表现。它不面向真实业务上线，而是给个人或小团队做方案验证：同一个问题交给不同流程处理，例如直接回答、先规划再回答、先生成再审查、两路方案再综合，最后比较答案质量、耗时、token 消耗和失败情况。

平台第一版应尽量轻，不引入复杂的工作流引擎。核心目标是把实验跑通，把结果记录下来，能让人复盘“为什么这个流程好一点或差一点”。

## 2. 主要用户

1. 做 LLM 应用原型的工程师，希望快速比较几种提示词组织方式。
2. 做内容生产或知识问答的产品人员，希望知道多一步审查是否值得。
3. 做模型评测的人，希望保留中间过程，便于人工判断。

## 3. 支持的流程

第一版支持四类流程：

- **Direct**：一个模型调用直接回答。
- **Plan-Then-Answer**：先让模型写计划，再让模型按计划回答。
- **Draft-Review-Revise**：先生成草稿，再审查草稿，最后根据审查意见修订。
- **Two-View Synthesis**：从两个不同角度生成答案，再综合成一个最终版本。

这些流程不需要做成真正的智能体系统。第一版可以把每个步骤看成一次模型调用，把输入、输出和耗时记录清楚即可。

## 4. 任务设计

任务文件可以采用 JSONL。每行一个任务，包含：

- `id`：任务编号。
- `category`：任务类型。
- `input`：任务内容。
- `reference`：可选参考材料。
- `expected_shape`：可选，描述理想答案的结构。

任务类型建议覆盖四类：

1. 简单解释：测试直接回答是否足够。
2. 复杂方案：测试规划步骤是否有帮助。
3. 方案审查：测试审查和修订是否能发现遗漏。
4. 取舍判断：测试多角度综合是否比单路回答更稳。

## 5. 输出记录

每次运行生成一个 `run_id`。每条任务结果记录为 JSON，包含：

- 任务信息。
- 使用的流程名称。
- 模型名称。
- 最终答案。
- 中间步骤输出。
- token 用量。
- 耗时。
- 是否成功。
- 错误信息。

结果可以先写到本地 JSONL 文件。后续再考虑写 SQLite。

## 6. 页面设计

第一版本地页面包括三块：

1. **运行页**：选择任务、选择流程、启动实验。
2. **结果页**：查看历史运行，点击单条记录看最终答案和中间过程。
3. **配置页**：填写模型名称、API 地址、温度、最大输出长度和价格。

页面不需要复杂权限，也不需要部署到服务器。能在本地浏览器打开即可。

## 7. 评估方式

第一版先记录基础指标：

- 平均耗时。
- 平均 token。
- 平均成本。
- 成功数量。
- 错误数量。

质量评价可以先人工看。后续可增加模型评分，但模型评分不能完全替代人工判断。

## 8. 第一版边界

- 不做多人协作。
- 不做复杂任务依赖。
- 不接外部工具调用。
- 不做自动提示词优化。
- 不做模型微调。
- 不强制接数据库。

先保证同一批任务可以稳定跑四种流程，并能保留足够的日志。
